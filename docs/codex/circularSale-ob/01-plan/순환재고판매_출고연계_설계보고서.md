# 순환재고 판매-출고 연계 설계보고서

## 1. 목적
- 순환재고 판매 생성 시 요청/승인 단계를 제거하고 즉시 출고로 연계한다.
- 판매 내역에서 출고 진행 상태를 직접 확인할 수 있도록 상태를 단순화한다.
- 판매 생성/조회 API와 출고 상태 동기화 규칙을 정립한다.

## 2. 범위
- 순환재고 판매 생성 API
- 순환재고 판매 내역 조회 API(목록/상세)
- 판매 생성 시 출고 헤더/아이템/상태이력 동시 생성 오케스트레이션
- 출고 상태 변경 시 판매 상태 동기화
- 소재(ESG) 스냅샷 저장 구조

## 3. 상태 재설계
### 3.1 판매 상태
- `READY_TO_SHIP`
- `IN_TRANSIT`
- `ARRIVED`

기존 `REQUESTED`, `CONFIRMED`, `CANCELLED`, `COMPLETED`는 사용하지 않는다.

### 3.2 시각 컬럼
- `completed_at`: 배송 완료(`ARRIVED`) 전이 시각으로 사용
- `confirmed_at`, `cancelled_at`, `cancel_reason`: 이번 범위에서 제외

## 4. 테이블/모델 설계
### 4.1 CIRCULAR_SALE_HEADER
- 판매 요약 데이터, 상태, 총계 수치 저장
- 출고 매핑 컬럼(`outbound_header_id`) 유지
- 상태는 3개 enum으로 고정

### 4.2 CIRCULAR_SALE_ITEM
- 재고 차감 단위 라인 저장
- SKU/상품명/카테고리/중량/수량/단가/금액 스냅샷 유지

### 4.3 CIRCULAR_SALE_STATUS_HISTORY
- 상태 변경 이력 저장
- `from_status` 추가(권장)로 전이 추적 강화

### 4.4 CIRCULAR_SALE_ITEM_MATERIAL (신규)
- ESG 대응용 소재 스냅샷 저장
- 권장 컬럼: `sale_item_id`, `material_code`, `material_name`, `ratio`, `sort_order`
- SKU 마스터 변경 이후에도 과거 판매 라인의 소재 정보 불변 보장

## 5. API 설계
### 5.1 판매 생성
- `POST /api/hq/circular-inventory/sales`
- 처리:
1. 요청 검증(필수값/음수/중복 inventoryId)
2. 재고 락(`PESSIMISTIC_WRITE`) 획득
3. 재고 차감
4. 판매 헤더/아이템/상태이력 생성
5. 출고 헤더/아이템/상태이력 생성(초기 상태 `READY_TO_SHIP`)
- 응답:
1. `saleId`, `saleNo`, `status`
2. 합계 수치
3. 핵심 품목 요약

### 5.2 판매 목록 조회
- `GET /api/hq/circular-inventory/sales`
- 필터:
1. 기간(from~to)
2. 거래처
3. 소재 구분
4. 키워드(판매번호 등)
- 정렬/페이징:
1. 기본 정렬 `soldAt desc, id desc`
2. `page`, `size`, `totalPages`, `totalElements`, `hasNext`, `hasPrevious`, `content`

### 5.3 판매 상세 조회
- `GET /api/hq/circular-inventory/sales/{saleId}`
- 반환:
1. 헤더 정보
2. 상세 품목
3. 합계 요약
4. 라인별 소재 스냅샷

## 6. 오케스트레이션/트랜잭션
- 판매 생성과 출고 생성은 단일 트랜잭션에서 처리한다.
- 중간 실패 시 전체 롤백한다.
- 재고는 비관적 락으로 oversell을 방지한다.

## 7. 출고 상태 동기화 설계
- 연동 지점: 기존 `WhOutboundService` 상태 전이 API
- 규칙:
1. 출고 `IN_TRANSIT` 전이 시 판매 상태 `IN_TRANSIT` 동기화 + 판매 상태이력 적재
2. 출고 `ARRIVED` 전이 시 판매 상태 `ARRIVED` 동기화 + `completed_at` 기록 + 판매 상태이력 적재
- 연결 키:
1. `wh_outbound_header.source_type = CIRCULAR_SALE`
2. `source_ref_no` 또는 `source_ref_id`로 판매 헤더 매핑

## 8. 검증 규칙
- 공통:
1. 필수값 누락 금지
2. 음수/0 허용 여부는 도메인 규칙대로 검증
3. 상태값 유효성 검증
- 판매 생성:
1. 재고 부족 시 실패
2. 요청 라인 중복 inventoryId 금지
3. 순환재고 상태가 아닌 inventory 차감 금지
- 상태 동기화:
1. 허용되지 않은 전이 차단
2. 동일 상태 재요청 처리 정책 일관화(no-op 또는 예외)

## 9. 인덱스/정합성
- 유니크:
1. `circular_sale_header.sale_no`
2. `circular_sale_item (sale_header_id, inventory_id)`
- 인덱스:
1. header: `(status, sold_at desc)`, `(buyer_id, sold_at desc)`, `(warehouse_id, sold_at desc)`
2. item: `(sale_header_id)`, `(sku_id)`, `(inventory_id)`
3. history: `(sale_header_id, changed_at desc)`
- 체크 제약:
1. 수량/중량/단가/금액 음수 금지

## 10. 테스트 전략
1. 생성 성공: 판매+출고+양쪽 상태이력 최초행 생성 검증
2. 롤백: 출고 생성 실패 시 판매 데이터 롤백 검증
3. 재고 검증: 부족/중복/음수 실패 검증
4. 조회 정합: 목록 필터/정렬/페이징 및 상세 필드 검증
5. 상태 동기화: 출고 전이에 따른 판매 상태/완료시각 반영 검증
6. 소재 불변성: SKU 마스터 변경 후 과거 판매 상세 소재 스냅샷 유지 검증

## 11. 범위 제외
- 판매 취소 기능 및 취소 상태
- 결제/정산 상태 관리

## 12. 재고 반영 시점 명시 (중요)
### 12.1 공통 원칙
- 실재고(`quantity`)와 가용재고(`available_quantity`)를 분리해 관리한다.
- 생성/승인 시점에는 **가용재고를 즉시 차감(예약)** 한다.
- 실재고 차감은 출고 확정(`IN_TRANSIT`) 시점에 수행한다.

### 12.2 흐름별 반영 규칙
1. 매장 발주 -> 출고
- 발주 승인 오케스트레이션에서 `reserveForOutboundUpTo(...)`로 가용재고를 즉시 예약 차감한다.
- 출고 확정 시 `moveReservedToInTransit(...)`로 reserved -> inTransit 전이한다.

2. 재고 이동(WH_TRANSFER) -> 출고
- 이동 실행 시 `reserveForOutboundUpTo(...)`로 출발 창고 가용재고를 즉시 예약 차감한다.
- 출고 확정 시 `moveReservedToInTransit(...)`로 reserved -> inTransit 전이한다.

3. 순환재고 판매 -> 출고
- 판매 생성 시 `reserveUpTo(...)`로 가용재고를 즉시 예약 차감한다.
- 출고 확정 시 `moveReservedToInTransitByInventoryId(...)` 후 `decreaseQuantityByInventoryId(...)`로 실재고를 차감한다.

## 13. 도메인 경계 명시
- 순환재고 판매의 목적지는 `CIRCULAR_BUYER`이며, 매장 입고 도메인과 연계하지 않는다.
- 즉, 순환판매는 `판매 -> 출고`까지만 오케스트레이션하며 `입고(RECEIVED)` 단계는 없다.
