# 순환판매-출고 리팩터링 계획 보고서

## 1. 목적/배경
- 순환재고 판매의 재고 반영 코드를 매장발주/재고이동 패턴과 정렬해 도메인 간 의미/흐름 일관성을 높인다.
- 출고 확정 로직에서 소스별 예외 분기를 최소화해 유지보수성과 확장성을 높인다.
- API 스펙/DB 대수정 없이 서비스 계층 정렬을 우선 적용한다.

## 2. Before / After
### Before
- 순환판매 생성 예약 처리: `CircularSaleService`에서 엔티티(`Inventory.reserveUpTo`) 직접 호출.
- 순환판매 출고확정 처리: `WhOutboundService.confirm()` 내부에 `CIRCULAR_SALE` 분기 + 전용 헬퍼로 `reserved -> in_transit`, `quantity` 차감 수행.

### After
- 순환판매 생성 예약 처리: `InventoryService.reserveForOutboundUpToByInventoryId(...)` 경유로 통일.
- 순환판매 출고확정 처리: `WhOutboundService.confirm()`은 공통 진입 `moveReservedStockForConfirm(...)` 사용.
  - `CIRCULAR_SALE`: `InventoryService.moveReservedToInTransitAndDecreaseByInventoryId(...)`
  - 그 외 소스: `InventoryService.moveReservedToInTransit(...)`
- 재고 반영 책임을 `InventoryService`에 집중시켜 도메인 서비스의 직접 엔티티 조작을 축소.

## 3. 재고 반영 시점 규약
- 공통 원칙
  - 생성/승인/실행 단계: `available 감소 + reserved 증가` (예약)
  - 출고 확정 단계: `reserved -> in_transit` 전이
- 순환판매 특이점
  - 출고 확정 단계에서 `quantity(실재고)`도 동시 차감
  - 즉, 순환판매는 출고 확정 후 `available/quantity` 수치가 같은 기준으로 정렬된다.

## 4. 오케스트레이션 시퀀스
1. 순환판매 생성 API
   - 요청 검증
   - 재고 예약(`reserveForOutboundUpToByInventoryId`)
   - 판매 헤더/아이템/상태이력 + 출고 헤더/아이템/상태이력 생성
2. 출고 확정 API
   - 소스타입별 재고 반영 전략 실행(`moveReservedStockForConfirm`)
   - 출고 상태 `READY_TO_SHIP -> IN_TRANSIT`
   - 순환판매 상태 동기화 `READY_TO_SHIP -> IN_TRANSIT`
3. 출고 도착 API
   - 출고 상태 `IN_TRANSIT -> ARRIVED`
   - 순환판매 상태 동기화 `IN_TRANSIT -> ARRIVED`
   - `completed_at` 반영

## 5. 참조 키/연결 규약
- `wh_outbound_header.source_type = CIRCULAR_SALE`
- `source_ref_no = sale_no`
- `source_ref_id = circular_sale_header.id`
- `wh_outbound_item.source_line_ref_id = circular_sale_item.id`
- 순환판매 confirm 시 `source_line_ref_id -> circular_sale_item.inventory_id` 경로로 정확한 CIRCULAR 재고 row를 식별한다.

## 6. 테스트 체크리스트
- 순환판매 생성
  - 예약 반영: `available 감소`, `reserved 증가`
  - 판매/출고/상태이력 생성 정합성
- 출고 확정
  - `reserved -> in_transit` 전이
  - `quantity` 차감
  - 판매 상태 `IN_TRANSIT` 동기화
- 출고 도착
  - 판매 상태 `ARRIVED`
  - `completed_at` 저장
- 회귀
  - 매장발주/재고이동 출고확정 경로 영향 없음

## 7. 리스크 / 롤백 포인트
- 리스크
  - 기존 순환판매에서 엔티티 직접 조작에 의존한 숨은 부작용이 있을 수 있음
  - confirm 단계 sourceLineRefId 누락 데이터가 있으면 재고 반영 실패
- 롤백 포인트
  - `WhOutboundService.confirm()`의 재고 반영 진입점(`moveReservedStockForConfirm`)만 이전 방식으로 복구 가능
  - `InventoryService` 추가 메서드 사용부를 기존 호출로 되돌리는 부분 롤백 용이
