# 순환재고 판매-출고 연계 구현보고서

## 1. 구현 요약
- 순환재고 판매 생성 시 요청/승인 단계를 제거하고 즉시 `READY_TO_SHIP`으로 생성되도록 구현했다.
- 판매 생성 트랜잭션 내에서 판매/출고/양쪽 상태이력을 동시 생성하도록 오케스트레이션을 구현했다.
- 출고 상태 전이(`IN_TRANSIT`, `ARRIVED`) 시 판매 상태를 동기화하도록 연동했다.
- 판매 상세에 ESG 대응용 소재 스냅샷을 포함할 수 있도록 모델을 추가했다.

## 3. 구현 상세
### 3.1 API
1. `POST /api/hq/circular-inventory/sales`
2. `GET /api/hq/circular-inventory/sales`
3. `GET /api/hq/circular-inventory/sales/{saleId}`

### 3.2 상태 모델
- 판매 상태 enum을 `READY_TO_SHIP`, `IN_TRANSIT`, `ARRIVED`로 구성했다.
- `completed_at`은 `ARRIVED` 전이 시점에 기록하도록 구현했다.

### 3.3 판매 생성 오케스트레이션
1. 요청/라인 검증 수행
2. 재고 락(`PESSIMISTIC_WRITE`) 획득
3. 재고 차감
4. 판매 헤더/아이템/상태이력 생성
5. 출고 헤더/아이템/상태이력 생성
6. 초기 상태를 양쪽 모두 `READY_TO_SHIP`으로 저장

### 3.4 출고 상태 동기화
- `WhOutboundService`의 상태 변경 처리 흐름에 순환판매 동기화를 연결했다.
1. 출고 `IN_TRANSIT` 전이 시 판매 `IN_TRANSIT` 반영 + 판매 상태이력 저장
2. 출고 `ARRIVED` 전이 시 판매 `ARRIVED` 반영 + `completed_at` 저장 + 판매 상태이력 저장

### 3.5 소재(ESG) 처리
- 판매 라인 소재 스냅샷 테이블(`CircularSaleItemMaterial`)을 추가했다.
- SKU 마스터 변경 이후에도 판매 시점 소재 정보가 유지되도록 설계했다.
