# 도메인 공통 코딩규칙/컨벤션 (Backend)

## 1. 목적
- 매장 판매 기능 구현 과정에서 합의한 설계/구현/코딩 스타일을 공통 규칙으로 고정한다.
- 이후 발주, 입고, 물류창고 출고, 순환재고 등 신규 도메인에도 동일 기준을 적용한다.

## 2. 아키텍처 원칙
- 기본 구조: `Controller -> Service -> Repository -> Entity`
- 패턴: MVC + 레이어드 아키텍처
- 응답 포맷: `BaseResponse` 기본 사용
- 컨트롤러 공개 메서드와 서비스 공개 메서드는 1:1 대응을 기본으로 한다.
- 서비스 내부 로직은 `private helper`로 역할을 분리한다.

## 3. 패키지/파일 구조
- 도메인 기준 패키지로 구성한다. (예: `...store.sale`)
- 도메인 기본 파일 세트
  - `XXXController`
  - `XXXService`
  - `XXXRepository`
  - `model/dto/XXXDto`, `model/entity/XXX`, `model/entity/XXXStatus`

## 4. DTO 규칙
- DTO는 도메인별 단일 파일 + `static inner class` 구조를 사용한다.
- 네이밍은 기능 중심으로 명확하게 작성한다.
  - 예: `SaleReq`, `SaleRes`, `SaleListRes`, `SaleDetailRes`, `SaleLineReq`, `SaleLineRes`
- Req/Res는 한 기능 단위로 인접 배치한다.
- 변환 메서드 규칙
  - `DTO -> Entity`: `toEntity(...)`
  - `Entity -> DTO`: `from(...)`
- `toEntity`는 DTO 자신의 값에 `this.field`를 사용한다.
- DTO 단독으로 표현 불가한 값은 `...Context` 객체로 전달한다.

## 5. Entity/JPA 규칙
- `@Entity`, `@Table`은 명시한다.
- `@Column(name = "...")`은 명시를 기본으로 한다.
- 상태값은 enum + `@Enumerated(EnumType.STRING)`을 사용한다.
- 기본 생성자는 `@NoArgsConstructor(access = PROTECTED)`를 사용한다.
- 인덱스/유니크/체크 제약은 애너테이션보다 SQL 마이그레이션을 기준으로 관리한다.

## 6. SQL/마이그레이션 규칙
- DDL, 제약, 인덱스는 SQL 마이그레이션 파일로 관리한다.
- MVP 단계에서도 조회/정합성에 필요한 최소 인덱스는 선반영한다.
- 엔티티 자동 생성에 의존하지 않고, 운영 기준 스키마는 SQL을 단일 진실원으로 둔다.

## 7. 컨트롤러 규칙
- 메서드명은 역할이 바로 보이게 작성한다.
  - 생성: `create`
  - 목록: `list` 또는 `findAll`
  - 상세: `detail`
- 주석은 단계형으로 통일한다.
  - 1. 요청 수신
  - 2. 서비스 호출
  - 3. 응답 반환

## 8. 서비스 규칙
- 공개 메서드는 비즈니스 단계를 순서대로 보이게 작성한다.
- 복잡한 로직은 helper 메서드로 분리한다.
  - 검증 helper
  - 조회 helper
  - 계산 helper
  - 매핑 helper
- 과도한 매핑 helper는 지양하고, 가능하면 메인 흐름에서 `DTO.from(...)`으로 반환한다.
- 내부 helper 주석 규칙
  - 시작 주석에 `사용하는 메서드: create, findAll, ...`를 명시한다.
  - 다음 줄에 helper 역할을 간결히 설명한다.

## 9. 트랜잭션/동시성 규칙
- 생성/수정성 작업은 단일 트랜잭션으로 처리한다.
- 경쟁 자원(재고)은 비관적 락(`PESSIMISTIC_WRITE`)을 기본 적용한다.
- 권장 순서: 락 획득 -> 검증 -> 차감/반영 -> 저장
- 예외 발생 시 전체 롤백(부분 반영 금지)을 보장한다.

## 10. 식별자/번호 생성 규칙
- `max + 1` 방식은 금지한다. (동시성 충돌 위험)
- PK 확보 후 비즈니스 번호를 생성한다.
- 판매번호 규칙: `SALE-{yyyyMMdd}-{id 최소 5자리 zero-pad}`

## 11. 예외/에러코드 규칙
- 도메인별 코드 대역을 사용한다. (판매: `4500~4505`)
- 메시지는 한국어 기준으로 관리한다.
- 케이스를 명확히 분리한다. (없음/검증실패/재고부족 등)

## 12. 인코딩/문자열 규칙
- 한글 주석/문자열 수정은 인코딩 깨짐을 최우선으로 점검한다.
- 파일 수정 직후 깨짐 여부를 재확인한다.
- 깨짐 발견 시 추가 작업 전 즉시 복구한다.

## 13. 문서화 규칙
- 기능별 산출물은 `docs/codex/{domain}` 하위에 남긴다.
  - `01-plan`, `02-implementation`, `03-postman` 등
- 다음 기능에도 동일 템플릿/수준으로 문서화한다.

## 14. 테스트 기준
- 정상 플로우: 생성/목록/상세 정합성
- 실패 플로우: 빈 요청, 수량 오류, 미존재 코드, 재고 부족
- 동시성: 동일 SKU 동시 요청 시 oversell 방지 검증
- 식별자: `saleNo` 형식/유일성 검증
