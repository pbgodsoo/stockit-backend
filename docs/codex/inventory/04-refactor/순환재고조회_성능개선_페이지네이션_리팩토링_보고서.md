# 순환재고 조회 성능개선/페이지네이션 리팩토링 보고서

- 작성일: 2026-05-20
- 대상: HQ 순환재고 조회(`/api/hq/inventories/circular`) 및 연동 FE 페이지네이션
- 범위:
  - BE: 순환재고 조회 로직(InventoryService, InventoryRepository)
  - FE: 순환재고 목록 store 정렬/페이지 동작(circularStockInventory.js)

## 1. 배경 및 목표

순환재고 조회 화면에서 다음 이슈가 반복적으로 발생했다.

- 페이지 전환 시 응답이 느림
- 창고/소재 필터 적용 시 체감 지연 큼
- 페이지네이션 결과가 사용자가 기대하는 순서와 다르게 보이는 경우 발생

목표는 다음 2가지였다.

1. 페이지/필터 처리의 병목 제거(근본 성능 개선)
2. 서버 페이지네이션 결과와 FE 표시 순서의 일관성 확보

---

## 2. 기존 문제점 분석

### 2.1 BE 병목 구조

기존 `findCircularInventories`는 아래 흐름이었다.

1. `CIRCULAR` 재고 전체 조회
2. Java 메모리에서 SKU/상품/카테고리/소재 조합
3. Java 스트림으로 키워드/창고/소재 필터
4. Java에서 정렬
5. 마지막에 `subList`로 페이지 슬라이스

즉, **요청마다 전체 데이터 스캔 + 메모리 필터 + 메모리 정렬 + 메모리 페이지** 구조였다.

영향:

- 페이지 번호만 바뀌어도 전체 재계산
- 필터 조합이 복잡할수록 지연 증가
- 데이터 증가 시 선형적으로 악화

### 2.2 FE 페이지 표시 문제

FE store(`circularStockInventory.js`)에서 서버 응답 rows를 다시 정렬하고 있었다.

- 서버가 페이지/정렬 결과를 내려줘도 FE에서 재정렬
- 결과적으로 사용자 기준 “서버 페이지 순서”와 UI 표시 순서가 어긋날 가능성 발생

## 3. 개선 과정

## 3.1 1차 안정화(즉시 체감 개선)

### (A) FE 재정렬 제거

- 파일: `Stockit-FE/src/stores/hq/circularStock/circularStockInventory.js`
- 변경: `inventoryRows` computed에서 로컬 sort 제거
- 효과: 서버가 내려준 페이지 순서 그대로 렌더링

### (B) BE 임시 캐시 + 선필터

- 파일: `Stockit-BE/src/main/java/org/example/stockitbe/hq/inventory/InventoryService.java`
- 변경:
  - 동일 필터/정렬 요청에 대해 짧은 TTL 캐시 적용(15초)
  - 창고코드 조건을 가능한 앞단에 적용
- 효과:
  - 페이지 이동/동일 필터 재조회 체감 개선
  - 불필요 계산 일부 축소

주의: 이 단계는 근본 해결이 아니라 완화책이었다.

## 3.2 2차 근본 개선(DB 페이징/필터 전환)

핵심 전환: **메모리 기반 처리 -> DB 기반 페이지/필터 처리**

### (A) 순환재고 페이지 전용 쿼리 도입

- 파일: `Stockit-BE/src/main/java/org/example/stockitbe/hq/inventory/InventoryRepository.java`
- 추가 메서드:
  - `findCircularInventoriesPaged(...)`
- 적용 조건:
  - `inventory_status = 'CIRCULAR'`
  - 창고 타입/창고 코드
  - 가용재고 > 0
  - 키워드(상품코드/상품명/SKU/소재명)
  - 소재 구분(천연 단일/합성/혼방)
  - 소재명 + 최소 비율(minRatio)
- `countQuery` 포함으로 totalPages/totalElements 정확성 확보

### (B) 페이지 row 전용 projection 추가

- 파일 추가:
  - `CircularInventoryPageRow.java`
- 목적:
  - 페이지 본문에 필요한 컬럼만 조회하여 전송/매핑 비용 축소

### (C) 소재 구성 전용 projection + 일괄 조회

- 파일 추가:
  - `CircularInventoryCompositionRow.java`
- 추가 메서드:
  - `findCircularInventoryCompositionsByItemCodes(...)`
- 방식:
  - 페이지에 포함된 itemCode만 모아 소재 구성 한 번에 조회
  - N+1 회피

### (D) Service 로직 교체

- 파일: `Stockit-BE/src/main/java/org/example/stockitbe/hq/inventory/InventoryService.java`
- 변경 전:
  - 전체 조회 -> 메모리 필터/정렬/페이지
- 변경 후:
  1. Repository 페이지 쿼리 호출
  2. 현재 페이지 itemCode 소재구성 일괄 조회
  3. material type/price/weight/circularSalePrice 계산 후 DTO 조립
  4. DB page metadata로 응답 생성

### (E) 소재 그룹 필터 정규화

- FE 라벨과 DB/정책 값을 매핑:
  - `천연 단일 섬유 -> NATURAL_SINGLE`
  - `합성 섬유 -> SYNTHETIC`
  - `혼방 -> BLEND`

## 4. 결과

## 4.1 기능/정합성

- 페이지네이션 메타(page/size/totalElements/totalPages) 정합성 유지
- 필터(창고/소재/키워드) 결과 일관성 향상
- FE에서 서버 페이지 순서를 보존하여 UX 혼선 제거

## 4.2 성능

개선 포인트:

- 전체 재고를 매 요청마다 메모리 스캔하던 경로 제거
- DB가 필터/페이지를 선처리하여 응답 대상 데이터량 감소
- 소재 구성 조회를 페이지 단위 일괄 조회로 제한

예상 효과(데이터 증가 구간):

- 페이지 전환 지연 감소
- 창고/소재 필터 응답 시간 감소
- 동시 사용자 환경에서 서버 메모리/CPU 부담 완화

---

## 5. 리스크/한계 및 후속 제안

1. 현재 정렬 키 지원
- DB 쿼리 정렬은 우선 `skuCode/quantity` 중심으로 반영
- `materialKgPrice`, `weight`, `circularSalePrice`까지 완전 SQL 정렬 이관 시 추가 개선 가능

2. 인덱스 점검
- 실제 운영 데이터 규모에서 `inventory`, `product_sku`, `product_master`, `product_material_composition`, `material`, `infrastructure` 조인 경로 인덱스 점검 필요

3. 캐시 정책
- DB 기반 전환 후 기존 임시 캐시는 의존도가 낮아졌으므로 단순화/제거 여부 판단 가능

## 7. 결론

이번 리팩토링은 순환재고 조회의 핵심 병목(전체 메모리 처리)을 제거하고, 페이지네이션 책임을 DB로 이관해 구조적으로 확장 가능한 형태로 바꾼 작업이다.

즉, “일시적인 캐시 완화”가 아니라 “조회 아키텍처 자체 전환”에 해당하며, 페이지 전환/필터 체감 성능과 표시 일관성 모두 개선하는 방향으로 정리됐다.
