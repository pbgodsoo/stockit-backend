# 매장 발주 배치 — Spring Batch 단계별 도입 계획

## 배경

현재 `StoreOrderBatchApproveService`는 `@Scheduled` + for-loop 구조로 발주 자동 승인 배치를 처리하고 있음.  
확인된 7가지 문제를 **한 번에 전부 고치지 않고**, 단계를 쪼개어 하나씩 해결하면서 중간중간 성능·안정성 변화를 k6로 측정·기록하는 것이 목표.  
최종적으로 Spring Batch 서버는 별도 MSA(`stockit-batch`)로 분리 예정.

### Stage 1 — Spring Batch 기본 + 이력 저장 + 중복 방지

**해결 문제**: 문제 1, 문제 3

**변경 내용:**

1. `build.gradle`에 의존성 추가

   ```gradle
   implementation 'net.javacrumbs.shedlock:shedlock-spring:6.x'
   implementation 'net.javacrumbs.shedlock:shedlock-provider-jdbc-template:6.x'
   ```

2. Spring Batch 메타 테이블 자동 생성 설정
   ```yaml
   # application.yml
   spring:
     batch:
       jdbc:
         initialize-schema: always
   ```

     `BATCH_JOB_EXECUTION_CONTEXT`, `BATCH_STEP_EXECUTION`, `BATCH_STEP_EXECUTION_CONTEXT` 6개 테이블 자동 생성

   ```sql
   CREATE TABLE shedlock (
     name VARCHAR(64) NOT NULL,
     lock_until TIMESTAMP NOT NULL,
     locked_at  TIMESTAMP NOT NULL,
     locked_by  VARCHAR(255) NOT NULL,
     PRIMARY KEY (name)
   );
   ```

4. `StoreOrderBatchApproveJob` 정의 — **Tasklet 방식** (기존 로직 그대로 래핑)

   ```
     └─ Step 1 (Tasklet): 기존 runAutoDaily() 로직 그대로 이동
   ```

**기존 로직 유지:** `StoreOrderBatchApproveItemService.approveOne()` 그대로 호출 (로직 변경 없음)

**예상 이슈:**

### Stage 3 — Partitioned Step (지역창고 단위 병렬 처리)

**해결 문제**: 문제 6 (순차 처리 → 병렬) ← **핵심 성능 개선**

**파티셔닝 기준: 지역창고 (Regional Warehouse Cluster)**

재고 락(`PESSIMISTIC_WRITE`)이 창고 단위로 걸리기 때문에 **같은 창고에 속한 매장이 서로 다른 파티션에 쪼개지면 DeadLock이 발생함**.
지역 단위로 창고 파티션을 나누면 한 파티션이 해당 지역의 창고들을 단독으로 처리하므로 교차 락이 발생하지 않음.

> **파티셔닝 기준을 결정하는 핵심 질문:**  
> "병렬로 처리했을 때 서로 충돌하지 않는 독립적인 단위가 무엇인가?"
>
> → 충돌 지점: `InventoryService.reserveForOutboundUpTo()`의 `PESSIMISTIC_WRITE`  
> → 락이 걸리는 단위: 창고(warehouse)별 재고  
> → 따라서 독립적인 단위 = **지역창고** → 파티셔닝 기준 = **지역창고(각 지역별 2개)**

재고 락(`PESSIMISTIC_WRITE`)이 창고 재고 단위로 걸리기 때문에, **같은 창고가 서로 다른 파티션(스레드)에 쪼개지면 DeadLock이 발생함**.
파티션 내부는 단일 스레드 순차 처리이므로, 같은 파티션 안에서는 창고가 여러 개이든 매장이 여러 개이든 데드락이 발생하지 않음.

**DeadLock 발생 조건 (유일)**: 같은 창고의 매장이 서로 다른 파티션에 쪼개질 때
```
매장별 파티션 (잘못된 설계) → DeadLock 발생:
  파티션 1 → 매장 A → 창고 W1 재고 락
  파티션 2 → 매장 B → 창고 W1 재고 락  ← 동일 창고 충돌!
  → 두 스레드가 W1 재고를 동시에 잠그려 함 → DeadLock

지역창고별 파티션 (올바른 설계) → DeadLock 없음:
  파티션 1 → 지역창고 R1 (창고 W1·W2 = 서로 백업 관계) → W1·W2 속한 매장 전담
  파티션 2 → 지역창고 R2 (창고 W3·W4 = 서로 백업 관계) → W3·W4 속한 매장 전담
  → 각 파티션이 담당 창고만 단독 접근 → 교차 락 없음
```

**핵심 규칙**: 창고 1개는 반드시 파티션 1개에만 속해야 함.  
파티션 1개에 창고가 여러 개 들어가는 것은 괜찮음 (순차 처리이므로).

**백업 창고와 파티셔닝:**
- 창고 22개는 지역 11개로 나뉘며, 각 지역의 창고 2개는 서로 PRIMARY↔BACKUP 관계
- 한 지역의 2개 창고를 **같은 파티션에 묶으면** → 한 스레드가 순차 처리 → DeadLock 없음
- 지역창고 단위 11개 파티션이 정확히 이 구조에 부합

```
도메인 구조:
  지역창고(파티션) 1 → [W1(PRIMARY), W2(BACKUP)] → W1·W2 담당 매장 ~10개
  지역창고(파티션) 2 → [W3(PRIMARY), W4(BACKUP)] → W3·W4 담당 매장 ~10개
  ...
  지역창고(파티션) 11 → [W21, W22] → W21·W22 담당 매장 ~10개
  → 11개 파티션 병렬 실행, 파티션 간 창고 겹침 없음 → DeadLock 없음
```

**도메인 현황:**
- 매장 110개, 창고 22개, 지역 11개
- 창고 2개 = 지역창고묶음 1개 (서로 PRIMARY↔BACKUP)
- 매장 1개 → PRIMARY 창고 1개 (`store_warehouse_map` role='PRIMARY')
- 지역창고(파티션) 1개 → 매장 약 10개 담당

**변경 내용:**

1. Master-Worker 파티션 구조
   ```
   Job: storeOrderBatchApproveJob
     ├─ Master Step (Partitioner)
     │    └─ regionWarehousePartitioner
     │         → 11개 지역창고묶음 × 파티션 1개 (지역창고당 창고 2개, 매장 ~10개씩)
     │           StepExecutionContext: { regionId, warehouseIds: [W1, W2] }
     └─ Worker Step (각 파티션별 Chunk)
          ├─ Reader: StepExecutionContext의 warehouseIds
          │          → store_warehouse_map JOIN
          │          → storeId 목록 조회
          │          → store_order_header WHERE store_id IN (...) AND status='REQUESTED'
          ├─ Processor: 승인 검증 (PRIMARY/BACKUP 창고 포함)
          └─ Writer:    approveOne() 호출
   ```

2. `ThreadPoolTaskExecutor` 설정
   ```java
   corePoolSize  = 11   // 지역창고 파티션 수에 맞춤
   maxPoolSize   = 11
   queueCapacity = 11   // 파티션 수만큼 큐 확보 (0으로 설정 시 조기 종료 위험!)
   ```
   - **queueCapacity=0 주의**: 파티션 수 > corePoolSize이면 `RejectedExecutionException` 발생
   - 해결: `queueCapacity = gridSize` 또는 `CallerRunsPolicy` 적용

3. HikariCP 커넥션 계산
   - Stage 3 진입 시: 10 → **30 connections/pod** 로 증설
   - 필요 커넥션 계산: (파티션 11개 × 2) + 2(JobRepository 등) + API 버퍼 = 약 30개
   - RDS db.t4g.small max_connections: 기본 68 → parameter group으로 **150 조정** (BATCH-16 완료)
   - Stage 5 MSA 분리 시: stockit-be=10/pod, stockit-batch=30/pod 로 분리

**예상 이슈:**
- `queueCapacity=0` 설정 시 성능 측정에서 갑자기 빠른 완료처럼 보이지만 실제로는 처리 누락
- DeadLock: 지역창고묶음별 파티션이면 원천 방지. 단, 파티션 설계 실수로 같은 창고가 다른 파티션에 쪼개지면 여전히 발생 가능

**테스트 방법:**
- k6 성능 측정 (지역 기준 창고 파티션 11개 기준 처리 시간 비교)
- 처리 건수 검증: `SELECT status, COUNT(*) FROM store_order_header GROUP BY status`
- Grafana DB connection 모니터링 (병렬 처리 시 spike 확인)
- DeadLock 발생 여부 로그 확인 (창고별이면 발생 없어야 함)

**기록할 포인트:**
- 파티션 수별 처리 시간 비교 표
- DB 커넥션 수 Grafana 캡처

### Stage 4 — faultTolerant + retry/skip 정책

**해결 문제**: 문제 7, 문제 2 보강

**변경 내용:**

1. Chunk에 `faultTolerant()` 적용
   ```java
   stepBuilder.chunk(chunkSize, transactionManager)
       .reader(reader)
       .processor(processor)
       .writer(writer)
       .faultTolerant()
       .retryLimit(3)
       .retry(OptimisticLockingFailureException.class)
       .retry(TransientDataAccessException.class)
       .noRetry(BaseException.class)               // 비즈니스 예외는 재시도 안 함
       .skipLimit(10)
       .skip(BaseException.class)                  // 비즈니스 예외는 스킵 후 계속
       .noSkip(DataIntegrityViolationException.class)
   ```

2. Skip 이벤트 리스너 추가
   ```java
   // StoreOrderBatchSkipListener.java
   // → skip된 orderNo, 사유를 store_order_batch_skip_log 테이블에 INSERT
   ```

3. 기존 `StoreOrderBatchApproveItemService.approveOne()` try-catch 제거
   → Spring Batch faultTolerant가 대신 처리

**예상 이슈:**
- `REQUIRES_NEW` 트랜잭션 + chunk rollback 상호작용:
  REQUIRES_NEW로 이미 커밋된 건은 chunk 롤백 시에도 되돌려지지 않음 → 멱등성 확인 필요
- skip된 건 재처리: JobParameter로 날짜 재지정 후 재실행 (Stage 2 덕분에 가능)

**테스트 방법:**
- 특정 orderNo에 예외 강제 유발
- Skip 이후 나머지 건 정상 처리 확인
- retry 로그 확인 (3회 재시도 후 skip)

**기록할 포인트:**
- retry/skip 로그 캡처
- `store_order_batch_skip_log` 테이블 데이터 확인

### Stage 5 — MSA 분리 (stockit-batch 서버)

**해결 문제**: 독립 배포, 배치 전용 스케일링

**변경 내용:**

1. 새 Spring Boot 프로젝트 `stockit-batch` 생성
   - 의존성: `spring-boot-starter-batch`, `spring-boot-starter-data-jpa`, MariaDB
   - `stockit-be`에서 이관할 코드:
     - `StoreOrderBatchApproveJob` (전체 Job/Step 정의)
     - `StoreOrderBatchApproveScheduler`
     - 필요한 Entity (`StoreOrderHeader`, `StoreOrderItem`, `Infrastructure`, `Inventory` 등)
     - Repository

2. `stockit-be` 변경
   - `POST /api/hq/store-orders/batch-approve/run` 유지 → 내부적으로 `stockit-batch` HTTP 호출로 위임

3. K8s 배포
   - **CronJob 방식 확정**: `stockit-batch` CronJob (replica=1)
   - `concurrencyPolicy: Forbid`, `backoffLimit: 0`, `restartPolicy: Never`
   - Job 완료 후 pod 자동 종료, ShedLock 불필요 (CronJob 자체가 중복 방지)

4. `JobRepository` 데이터소스 결정
   - **Option A (권장)**: 기존 MariaDB 공유 — 단순, t4g.micro 현황에서 비용 추가 없음
   - Option B: 별도 DB — 완전 분리, 추가 비용

**예상 이슈:**
- stockit-be ↔ stockit-batch 간 Entity 중복 관리 → 공통 모듈 분리 또는 API 기반 통신
- 네트워크 레이턴시: stockit-be에서 stockit-batch API 호출 시 추가 지연

**테스트 방법:**
- stockit-batch 독립 배포 후 k6 성능 측정
- stockit-be 재배포 없이 배치 코드 변경 가능한지 확인

## 단계별 성능 측정 계획

| Stage | 측정 조건 | 기록 항목 |
|-------|-----------|-----------|
| 0 (AS-IS) | 1,100건 / 2,200건 | 처리 시간, 힙 메모리 |
| 1 | 1,100건 / 2,200건 | 처리 시간 (변화 없어야 함), 이력 DB 확인 |
| 2 | 1,100건, chunk size 10/50/100/200 | 처리 시간, 힙 메모리 피크 |
| 3 | 1,100건, partition 5/10/20 | 처리 시간, DB 커넥션 수 |
| 4 | 일부 실패 inject | skip 건수, retry 로그 |
| 5 | 1,100건 (MSA 분리 후) | 처리 시간, 네트워크 오버헤드 |

## 관련 파일

| 파일 | 역할 |
|------|------|
| `StoreOrderBatchApproveService.java` | AS-IS 배치 로직 (→ Job/Step으로 이관 대상) |
| `StoreOrderBatchApproveItemService.java` | 건별 `REQUIRES_NEW` 트랜잭션 (Stage 4까지 재사용) |
| `StoreOrderBatchApproveScheduler.java` | `@Scheduled` → Stage 1에서 `@SchedulerLock` 추가 |
| `StoreOrderBatchApproveController.java` | 수동 API (→ JobLauncher 호출로 교체) |
| `build.gradle` | spring-boot-starter-batch, shedlock 추가 |
| `StoreOrderHeaderRepository.java` | Stage 2에서 Cursor 기반 쿼리 추가 필요 |
| `StoreOrderApprovalOrchestrationService.java` | 재고 예약 로직 (변경 없음) |

## 단계 진행 원칙

1. **한 단계 = 한 번의 k6 Before/After 측정**
2. 이슈 발생 시 → 해결 방법과 함께 기록
   - 참고: 타 팀 사례 — PagingItemReader OFFSET 밀림, queueCapacity=0 조기 종료, DeadLock
3. 각 단계 완료 후 Grafana 스크린샷 저장
4. Stage 1~4는 `stockit-be` 내에서 진행
5. Stage 5는 개발환경 검증 후 운영 배포
