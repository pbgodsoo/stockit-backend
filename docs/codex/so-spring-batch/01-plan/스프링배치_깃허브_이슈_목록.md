## Stage 1 — Spring Batch 기본 + 이력 저장 + 중복 방지

### [BATCH-1] Spring Batch 의존성 추가 및 기본 설정
**라벨:** `spring-batch` `infra`  
**작업 내용:**
- `build.gradle`: `spring-boot-starter-batch`, shedlock 의존성 추가
- `application.yml`: `spring.batch.job.enabled=false` 설정 (서버 시작 시 자동 실행 방지)
- `application.yml`: `spring.batch.jdbc.initialize-schema=always` 설정

**완료 기준:**
- 앱 정상 구동
- `BATCH_JOB_INSTANCE` 등 메타 테이블 6개 자동 생성 확인

---

### [BATCH-2] ShedLock 테이블 생성 및 스케줄러 중복 실행 방지 적용
**라벨:** `spring-batch` `infra`  
**작업 내용:**
- `shedlock` 테이블 DDL 실행
  ```sql
  CREATE TABLE shedlock (
    name       VARCHAR(64)  NOT NULL,
    lock_until TIMESTAMP    NOT NULL,
    locked_at  TIMESTAMP    NOT NULL,
    locked_by  VARCHAR(255) NOT NULL,
    PRIMARY KEY (name)
  );
  ```
- `StoreOrderBatchApproveScheduler`에 `@SchedulerLock` 추가
  ```java
  @SchedulerLock(name = "storeOrderBatchApproveJob", lockAtMostFor = "PT30M")
  ```

**완료 기준:**
- k6 VU=2 동시 실행 시 pod 1개만 실행되는 로그 확인
- 나머지 pod는 락 획득 실패 로그 출력

---

### [BATCH-3] Tasklet 기반 StoreOrderBatchApproveJob 구현
**라벨:** `spring-batch`  
**작업 내용:**

- 기존 `runAutoDaily()` 로직을 Tasklet으로 래핑 (로직 변경 없음)

**완료 기준:**

## Stage 3 — Partitioned Step (창고 단위 병렬)

### [BATCH-8] warehousePartitioner 구현
**라벨:** `spring-batch`  
**작업 내용:**
- `store_warehouse_map` (role='PRIMARY') 기반 창고별 파티션 분할 구현
- 파티션 내 warehouseId 목록 → storeId 목록 → `StepExecutionContext` 전달

**파티셔닝 핵심 규칙:**
> 창고 1개는 반드시 파티션 1개에만 속해야 함.  
> (같은 창고가 여러 파티션에 쪼개지면 PESSIMISTIC_WRITE DeadLock 발생)

**완료 기준:**
- 22개 창고가 N개 파티션으로 겹침 없이 분배되는지 단위 검증

---

**완료 기준:**
- 병렬 처리 로그 확인 (스레드 N개 동시 실행)
- 전체 건수 처리 누락 없음
- DeadLock 없음

## Stage 4 — faultTolerant + retry/skip

### [BATCH-11] faultTolerant retry/skip 정책 적용
**라벨:** `spring-batch`  
**작업 내용:**
- Chunk에 `faultTolerant()` 추가
  ```java
  .faultTolerant()
  .retryLimit(3)
  .retry(TransientDataAccessException.class)
  .retry(DeadlockLoserDataAccessException.class)
  .noRetry(BaseException.class)
  .skipLimit(20)
  .skip(BaseException.class)
  .noSkip(DataIntegrityViolationException.class)
  ```
- `StoreOrderBatchApproveService`의 기존 try-catch 제거

**완료 기준:**
- 예외 inject 테스트 → skip 후 나머지 건 정상 처리 확인
- retry 로그 확인 (3회 재시도 후 skip)
