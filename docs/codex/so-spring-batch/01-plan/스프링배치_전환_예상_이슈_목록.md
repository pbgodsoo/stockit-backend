# Spring Batch 전환 — 단계별 예상 이슈 목록

## 개요

Spring Batch 단계별 도입 과정에서 이 코드베이스 특성상 마주칠 가능성이 높은 이슈들을 정리.  
각 이슈는 발생 단계, 원인, 해결 방법을 함께 기록.

---

## Stage 1 — Spring Batch 기본 도입

### 이슈 1. Spring Boot 시작 시 Job 자동 실행

**증상:** 서버 재시작할 때마다 배치가 자동으로 실행됨  
**원인:** `spring-boot-starter-batch` 추가 시 기본값이 `job.enabled=true`  
**해결:**
```yaml
spring:
  batch:
    job:
      enabled: false  # 반드시 추가
```

---

### 이슈 2. JPA 트랜잭션 매니저 충돌

**증상:** 앱 구동 시 `NoUniqueBeanDefinitionException` 또는 트랜잭션 관련 오류  
**원인:** 기존 JPA `PlatformTransactionManager`와 Spring Batch 내부 트랜잭션 매니저 충돌  
**해결:** `BatchConfig`에서 트랜잭션 매니저를 명시적으로 지정
```java
@Bean
public Step approveStep(JobRepository jobRepository,
                        PlatformTransactionManager transactionManager) {
    return new StepBuilder("approveStep", jobRepository)
        .tasklet(approveTasklet(), transactionManager)  // 명시 지정
        .build();
}
```

## Stage 2 — Chunk 방식 + JobParameter

### 이슈 3. JpaCursorItemReader + @Transactional(REQUIRES_NEW) EntityManager 충돌

**증상:** `org.hibernate.HibernateException: Already a transaction in progress` 또는 영속성 컨텍스트 오류  
**원인:** Reader가 열어둔 JPA 영속성 컨텍스트 안에서 `approveOne()`이 `REQUIRES_NEW`로 새 트랜잭션을 열려 할 때 같은 EntityManager 충돌  
**해결:** `JdbcCursorItemReader`로 교체 (읽기는 JDBC, 처리는 기존 JPA 유지)
```java
// JpaCursorItemReader 대신
JdbcCursorItemReader<StoreOrderHeader> reader = new JdbcCursorItemReaderBuilder<StoreOrderHeader>()
    .dataSource(dataSource)
    .sql("SELECT * FROM store_order_header WHERE status = 'REQUESTED' ...")
    .rowMapper(new StoreOrderHeaderRowMapper())
    .build();
```

---

### 이슈 4. PagingItemReader OFFSET 밀림으로 처리 누락 (조용한 버그)

**증상:** 배치가 정상 완료됐는데 일부 발주건이 여전히 REQUESTED 상태로 남음  
**원인:** `JpaPagingItemReader` 사용 시, 처리 완료 건이 status=APPROVED로 바뀌면 다음 페이지 데이터가 앞으로 당겨지면서 일부 건이 Reader에서 누락됨  
```
OFFSET 0: [A, B, C, D, E] → A, B, C 처리 후 status 변경
OFFSET 3: [F, G, H, ...] → D, E가 앞으로 당겨졌으므로 D, E 누락
```
**해결:** `JpaCursorItemReader` 또는 `JdbcCursorItemReader` 사용 (커서 방식은 OFFSET 없음)  
**주의:** 인터넷 예제 대부분이 PagingItemReader를 사용하므로 팀 내 공유 필요

---

### 이슈 5. Spring Batch 5.x JobParameter 타입 변경

**증상:** `JobParameter` 관련 컴파일 에러 또는 런타임 타입 오류  
**원인:** Spring Boot 3.x는 Spring Batch 5.x를 사용하는데, JobParameter 생성 방식이 변경됨  
**해결:**
```java
// Boot 2.x / Batch 4.x 방식 (사용 불가)
new JobParametersBuilder()
    .addString("fromDate", "2026-05-24")

// Boot 3.x / Batch 5.x 방식 — 이 프로젝트 적용 파라미터
new JobParametersBuilder()
    .addLocalDateTime("fromDateTime", LocalDateTime.of(2026, 5, 24, 0, 0, 0))
    .addLocalDateTime("toDateTime",   LocalDateTime.of(2026, 5, 24, 23, 59, 59))
    .addString("runType", "MIDNIGHT")
    // MANUAL 시 추가:
    // .addString("runId", UUID.randomUUID().toString())
```
> 파라미터 구성: `fromDateTime`, `toDateTime` (LocalDateTime), `runType` (MIDNIGHT|MANUAL), `runId` (MANUAL 시만, UUID)

## Stage 3 — Partitioned Step (이슈 가장 많은 단계)

### 이슈 6. DeadLock — 매장별 파티션 시 발생, 지역창고별 파티션으로 원천 방지

**증상:** `com.mysql.jdbc.exceptions.jdbc4.MySQLTransactionRollbackException: Deadlock found when trying to get lock; try restarting transaction`  
**원인:** `InventoryService.reserveForOutboundUpTo()`에 `PESSIMISTIC_WRITE` 락이 창고 재고 단위로 걸림.  
**매장별 파티션**으로 설계하면 서로 다른 파티션이 **같은 창고의 재고**를 동시에 잠그려 해서 DeadLock 발생.

```
매장별 파티션 (잘못된 설계) → DeadLock:
  파티션 1 → 매장 A → 창고 W1 재고 락 시도
  파티션 2 → 매장 B → 창고 W1 재고 락 시도  ← 동일 창고!
  → 서로 대기 → DeadLock

지역창고별 파티션 (올바른 설계) → DeadLock 없음:
  파티션 1 → 지역창고 R1 (창고 W1·W2 = 서로 백업 관계) → W1·W2 속한 매장 전담
  파티션 2 → 지역창고 R2 (창고 W3·W4 = 서로 백업 관계) → W3·W4 속한 매장 전담
  → 각 파티션이 담당 창고만 단독 접근 → 교차 락 없음
```

**DeadLock 발생 조건 (유일):** 같은 창고의 매장이 **서로 다른 파티션(스레드)에 쪼개질 때**  
파티션 내부는 단일 스레드 순차 처리이므로, 같은 파티션 안에 창고가 여러 개 있어도 데드락 발생하지 않음.

**해결:** **파티셔닝 기준을 지역별 창고로 설계 — 창고 1개는 반드시 파티션 1개에만 속하도록 설계**

- 지역 11개 × 창고 2개(서로 PRIMARY↔BACKUP) = 파티션 **11개** (지역별 창고묶음 1개 = 파티션 1개)
- `store_warehouse_map` (role='PRIMARY') 기준으로 지역창고 단위 파티션 구성
- **절대 안 되는 것**: 같은 창고의 매장 일부는 파티션 1, 나머지는 파티션 2에 배정하는 것

### 이슈 7. queueCapacity=0 설정 시 파티션 조기 종료

**증상:** 배치가 빠르게 완료됐는데 처리 건수가 예상보다 적음. `RejectedExecutionException` 로그  
**원인:** `ThreadPoolTaskExecutor`의 `queueCapacity=0` 설정 시, 대기 큐가 없으므로 파티션 수 > corePoolSize이면 초과 파티션이 즉시 거절됨
```java
// 위험한 설정
executor.setCorePoolSize(4);
executor.setQueueCapacity(0);  // 파티션 10개면 6개 거절됨
```
**해결:**
```java
executor.setQueueCapacity(gridSize);  // 파티션 수만큼 큐 확보
// 또는
executor.setRejectedExecutionHandler(new CallerRunsPolicy());
```

### 이슈 8. 배치 실행 중 일반 API 커넥션 부족

**증상:** 배치 실행 중 일반 사용자 API 요청 타임아웃 발생  
**원인:** HikariCP pool 부족. 파티션 11개 동시 실행 시 커넥션 다수 점유 → 일반 트래픽용 커넥션 부족  
**현재 환경 (Stage 1~2):**
```
stockit-be pod 1: HikariCP 10개
stockit-be pod 2: HikariCP 10개
RDS db.t4g.small max_connections = 68 (기본값)
```
**해결:**
- **Stage 3 진입 전**: HikariCP 10 → **30 connections/pod** 로 증설
- 필요 커넥션 계산: (파티션 11개 × 2) + 2(JobRepository 등) + API 버퍼 = 약 30개
- RDS max_connections: parameter group으로 **150 조정** (BATCH-16 완료, 비용 변동 없음)
- Stage 5 MSA 분리 후: stockit-be=10/pod, stockit-batch=30/pod 로 분리

### 이슈 9. @StepScope 누락으로 스레드 간 상태 공유

**증상:** 서로 다른 파티션 데이터가 뒤섞이거나, 한 파티션의 처리가 다른 파티션 결과에 영향을 줌  
**원인:** 파티션 워커 Step의 Reader/Writer를 `@StepScope` 없이 싱글톤으로 등록하면 여러 스레드가 같은 인스턴스를 공유  
**해결:** Reader, Processor, Writer 모두 `@StepScope` 적용
```java
@Bean
@StepScope  // 반드시 추가
public JdbcCursorItemReader<StoreOrderHeader> partitionReader(
        @Value("#{stepExecutionContext['storeIds']}") List<Long> storeIds) {
    ...
}
```

## Stage 4 — faultTolerant + retry/skip

### 이슈 10. REQUIRES_NEW 커밋 건 + chunk 롤백 불일치

**증상:** `BATCH_STEP_EXECUTION`은 FAILED인데 DB에는 일부 건이 APPROVED 상태로 남음  
**원인:** `approveOne()`이 `REQUIRES_NEW`로 커밋된 건은 chunk 롤백 시에도 되돌아오지 않음. 재실행 시 이미 APPROVED된 건을 또 처리하려다 `validateRequestedOnly()` 예외 발생  
**해결:** Reader 조회 조건에 `status = REQUESTED`가 포함되어 있으므로 재실행 시 이미 APPROVED된 건은 자동으로 제외됨. 단, 이 멱등성이 유지되는지 확인 필요

---

### 이슈 11. skipLimit 초과 시 Job 전체 FAILED

**증상:** 소수의 실패 건 때문에 배치 전체가 중단됨  
**원인:** `skipLimit`을 너무 작게 설정하면 임계치 초과 시 `SkipLimitExceededException`으로 Job 전체 FAILED  
**해결:** 비즈니스 성격에 맞게 skipLimit 설정. 발주 건수 기준 1~2% 수준 권장
```java
.skipLimit(20)  // 1,100건 기준 약 2% 수준
```

## 전 단계 공통

### 이슈 14. Spring Batch 5.x API — 인터넷 예제와 다름

**증상:** 예제 코드 그대로 복붙 시 컴파일 에러  
**원인:** Spring Boot 3.x = Spring Batch **5.x** 사용. 인터넷 예제 대부분이 Batch 4.x 기준이며 주요 클래스가 변경/삭제됨

| Batch 4.x (예제) | Batch 5.x (실제 사용) |
|------------------|----------------------|
| `JobBuilderFactory` | 삭제 → `new JobBuilder("name", jobRepository)` |
| `StepBuilderFactory` | 삭제 → `new StepBuilder("name", jobRepository)` |
| `JobParameter(String)` | `JobParameter<>(value, Type.class)` |

### 이슈 15. 테스트 데이터 재삽입 반복 작업

**증상:** 각 Stage k6 테스트마다 수동으로 REQUESTED 데이터 INSERT 필요  
**원인:** 배치 실행 후 REQUESTED → APPROVED로 상태 변경되므로 재테스트 전 데이터 초기화 필요  
**해결:** 테스트 데이터 INSERT SQL 스크립트 파일로 관리
```sql
-- tools/test-data/insert_requested_orders.sql
-- 실행: SET SQL_SAFE_UPDATES = 0; → INSERT → SET SQL_SAFE_UPDATES = 1;
```

### 이슈 16. 수동 배치(MANUAL) 실행 중 MIDNIGHT 배치 실행 시 중복 처리

**증상:** MIDNIGHT 배치 실행 중 수동 배치 API를 호출하면 동일 날짜 범위를 중복 처리할 수 있음  
**원인:** ShedLock은 같은 이름의 Job 중복만 방지. 날짜 범위가 겹치는 MIDNIGHT+MANUAL 동시 실행은 차단 안 됨  
**해결:** 수동 배치 API 호출 시 `JobExplorer`로 실행 중 Job 상태 확인 후 차단
```java
// StoreOrderBatchApproveController (수동 배치 API)
Set<JobExecution> runningExecutions =
    jobExplorer.findRunningJobExecutions("storeOrderBatchApproveJob");

// 1. MIDNIGHT 배치 실행 중이면 409 반환
boolean midnightRunning = runningExecutions.stream()
    .anyMatch(e -> "MIDNIGHT".equals(e.getJobParameters().getString("runType")));
if (midnightRunning) throw new ConflictException("MIDNIGHT 배치 실행 중");

// 2. 날짜 범위 겹치는 MANUAL 배치 실행 중이면 409 반환
boolean rangeOverlap = runningExecutions.stream()
    .anyMatch(e -> isRangeOverlapping(e, requestFrom, requestTo));
if (rangeOverlap) throw new ConflictException("동일 날짜 범위 배치 실행 중");
```
> MIDNIGHT 배치의 경우 ShedLock이 단일 실행을 보장하므로 추가 체크 불필요.  
> 수동 배치만 위 로직 적용.
