# 매장 입고(Store Inbound) MVP 구현 내역 (2026-05-10)

## 1. 개요
- 매장 입고 조회/확정 API를 구현했다.
- 입고 확정 시 재고 반영, 입고 상태 이력 기록, 발주 완료 전이를 동일 트랜잭션으로 처리하도록 반영했다.

## 2. 구현 범위
- API
  - `GET /api/store/inbound`
  - `GET /api/store/inbound/{inboundNo}`
  - `POST /api/store/inbound/{inboundNo}/confirm`
- 도메인 처리
  - `PENDING_RECEIPT -> RECEIVED` 상태 전이
  - `StoreInboundStatusHistory(RECEIVED)` 저장
  - 매장 재고 증가 반영
  - 연계 입고 전건 수령 시 발주 `COMPLETED` + 발주 상태 이력 저장

## 4. 핵심 로직
- `StoreInboundService.confirm(...)`
  - 로그인 매장 범위 해석
  - 입고 소유권 검증
  - 상태 전이 가능 여부 검증
  - 입고 라인 기준 재고 증가
  - 입고 헤더 `RECEIVED` 전이
  - 입고 상태 이력 저장
  - 같은 발주의 연계 입고 전건 수령 여부 확인 후 발주 `COMPLETED` 전이
- `InventoryService.increaseOnHandAndAvailable(...)`
  - `(skuId, locationId, NORMAL)` 재고 row 락 조회
  - 존재 시 `quantity/availableQuantity` 동시 증가
  - 미존재 시 신규 row 생성

## 5. 계약/응답
- 공통 응답 래핑 `BaseResponse` 유지
- 발주 응답의 입고 파생 필드(`totalInboundCount`, `receivedInboundCount`, `inboundProgress`) 유지
- 입고 상세 응답에 연계 출고 요약 포함

## 6. 트랜잭션/롤백 결과
- 입고 확정 처리(`confirm`)는 단일 트랜잭션으로 동작
- 재고 반영/상태 전이/이력/발주 완료 전이 중 하나라도 실패 시 전체 롤백

## 7. 검증 현황
- 코드 레벨 검증
  - 서비스/컨트롤러/엔티티 반영 완료
- 빌드
  - 환경 의존 이슈(워크스페이스 기존 상태)로 전체 컴파일 변동 가능성 있음
- 시나리오 테스트
  - 정상/실패/분할입고 E2E 테스트는 별도 실행 필요

## 8. 남은 작업
- 매장 입고 전용 에러코드(`STORE_INBOUND_*`) 분리 여부 확정 및 반영
- Postman/E2E 시나리오 검증
- FE 연동 확인(입고 목록/상세/확정 동작)
