# 매장 판매 백엔드(v1) 구현 내역 정리

## 1. 구현 범위
- 포함: 판매 등록(C), 판매 목록 조회(R), 판매 상세 조회(R)
- 제외: 판매 분석, 취소/환불, 결제수단, 고객정보

## 2. 엔드포인트 구현 현황
- `POST /api/store/sales`
  - 판매 등록
  - 재고 검증 및 차감
  - 판매번호 생성 후 저장
- `GET /api/store/sales`
  - 판매 목록 조회
  - 매장코드/기간/키워드 필터 지원
- `GET /api/store/sales/{saleNo}`
  - 판매 상세 조회

## 3. 아키텍처/코드 구조
- 패턴: `Controller -> Service -> Repository -> Entity`
- 응답: `BaseResponse` 래핑 유지
- DTO: `StoreSaleDto` 내부 `static class` 구조 사용
  - `SaleReq`, `SaleRes`
  - `SaleListRes`, `SaleDetailRes`
  - `SaleLineReq`, `SaleLineRes`
  - `SaleHeaderContext`, `SaleLineContext`

## 4. 핵심 비즈니스 처리
### 4.1 판매 생성
- 요청 검증(아이템 비어있음/수량 오류)
- 매장 조회(STORE 타입)
- SKU/상품/카테고리 조회
- 재고 행 잠금(`PESSIMISTIC_WRITE`) 후 판매 가능 수량 검증
- 재고 차감 적용
- 헤더/아이템 저장
- 판매번호 부여

### 4.2 판매번호 규칙
- 규칙: `SALE-{yyyyMMdd}-{id 5자리 zero-pad}`
- 구현: 헤더 1차 저장으로 PK `id` 확보 후 `saleNo` 업데이트
- 특징: `max+1` 미사용으로 동시성 충돌 위험 제거

## 5. 동시성/트랜잭션
- `create` 메서드 단일 트랜잭션 처리
- 재고 조회 시 `InventoryRepository.findBySkuIdAndLocationIdForUpdate(...)` 사용
- 락 획득 후 검증/차감/저장 진행
- 예외 발생 시 전체 롤백(부분 반영 방지)

## 6. 재고 기준 정책 반영
- 판매 가능 판단: `availableQuantity`가 아닌 `quantity`(실재고) 기준
- 재고 차감도 `quantity` 중심으로 동작하도록 반영

## 7. DTO 변환 규칙 정리
- DTO -> Entity
  - `SaleReq.toEntity(SaleHeaderContext)`
  - `SaleLineReq.toEntity(SaleLineContext)`
- Entity -> DTO
  - `SaleRes.from(...)`
  - `SaleListRes.from(...)`
  - `SaleDetailRes.from(...)`
  - `SaleLineRes.from(...)`
- 서비스에서 DTO 변환 메서드를 직접 사용하도록 정리

## 8. 서비스 코드 스타일 정리 결과
- 컨트롤러 대응 공개 메서드 3개 유지
  - `create`, `findAll`, `findDetail`
- 내부 메서드는 helper로 분리
- 내부 메서드 주석 형식 통일
  - `사용하는 메서드: ...`
  - 메서드 역할 설명
- 과도한 매핑 helper(`toSaleRes`, `toSaleLineResList`) 제거
  - 메인 흐름 내에서 바로 `from(...)` 변환 후 반환

## 9. 데이터베이스/인덱스
- 판매 관련 테이블/제약/인덱스는 SQL 마이그레이션으로 관리
- 핵심 인덱스
  - `store_sale_header(store_id, sold_at)`
  - `store_sale_item(sale_header_id)`
  - `store_sale_item(sku_id)`
- 핵심 제약
  - `sale_no UNIQUE`
  - 수량/금액 유효성 제약

## 10. 예외 코드 정리
- 판매 도메인 예외 코드(4500~4505) 적용
  - 판매 없음, 빈 아이템, 수량 오류, 매장 없음, SKU 없음, 재고 부족
