# 도메인 공통 코딩 규칙/컨벤션

## 1. 목적
- 본 문서는 판매 기능 구현 과정에서 합의된 설계/구현 방식을 표준화한다.
- 이후 발주, 입고, 물류창고 출고, 순환재고 등 신규 도메인에도 동일하게 적용한다.

## 2. 아키텍처 원칙
- 기본 구조: `Controller -> Service -> Repository -> Entity`
- 패턴: MVC + 레이어드 아키텍처
- 응답 래퍼: `BaseResponse` 사용 유지
- 공개 서비스 메서드: 컨트롤러 엔드포인트와 1:1 대응
- 내부 로직: `private helper` 메서드로 책임 분리

## 3. 패키지/파일 구조 규칙
- 도메인 패키지 기준으로 구성한다. 예: `...store.sale`
- 도메인별 기본 파일 세트
  - `XXXController`
  - `XXXService`
  - `XXXRepository`(필요 시 다수)
  - `model/XXXDto`, `model/XXXEntity`, `model/XXXStatus`
- DTO는 도메인별 단일 파일에 inner class로 묶는다.

## 4. DTO 구현 규칙
- DTO 파일명: `도메인명 + Dto` (예: `StoreSaleDto`)
- DTO 클래스는 `static inner class`로 구성한다.
- 네이밍은 기능 중심으로 명확하게 한다.
  - 요청: `SaleReq`, `SaleLineReq`
  - 응답: `SaleRes`, `SaleListRes`, `SaleDetailRes`, `SaleLineRes`
- 요청/응답은 기능 단위로 인접 배치한다(탐색성 우선).
- DTO 변환 메서드 규칙
  - `DTO -> Entity`: `toEntity(...)`
  - `Entity -> DTO`: `from(...)`
- `toEntity`는 DTO 자신의 값은 `this.field`를 사용한다.
- DTO에 없는 외부 값은 `...Context` 객체로 전달한다.
  - 예: `SaleHeaderContext`, `SaleLineContext`

## 5. Entity 구현 규칙
- `@Entity`, `@Table` 명시
- 컬럼명은 `@Column(name = "...")` 명시를 기본으로 한다.
  - 이유: DB 스키마 명시성/일관성/리뷰 가독성 확보
- 인덱스/유니크 제약은 엔티티 주석 수준이 아니라 SQL 마이그레이션으로 확정 관리한다.
- 상태값은 enum 사용(`@Enumerated(EnumType.STRING)`).
- 기본 생성자는 `protected` 제한(`@NoArgsConstructor(access = PROTECTED)`).

## 6. Controller 구현 규칙
- 엔드포인트 메서드 네이밍 통일
  - 생성: `create`
  - 목록: `list` 또는 `findAll`(컨트롤러에서는 `list` 권장)
  - 상세: `detail`
- 흐름 주석 3단계 유지
  - 1. 요청 수신
  - 2. 서비스 호출
  - 3. 응답 반환
- 유효성 검증은 DTO 어노테이션 + 서비스 로직 검증을 병행한다.

## 7. Service 구현 규칙
- 서비스 공개 메서드(컨트롤러 대응)는 비즈니스 흐름을 단계형으로 작성한다.
- 내부 helper는 역할별로 분리한다.
  - 검증 helper
  - 조회 helper
  - 저장 helper
  - 필터/변환 helper
- 과도한 매핑 helper는 지양한다.
  - 가능하면 메인 흐름에서 `DTO.from(...)`으로 즉시 반환
- 내부 helper 주석 규칙
  - 첫 줄: `사용하는 메서드: create, findAll ...`
  - 둘째 줄: 해당 helper 역할 설명

## 8. 트랜잭션/동시성 규칙
- 생성/수정 계열은 단일 트랜잭션으로 처리한다.
- 재고 등 경합 자원은 비관적 락(`PESSIMISTIC_WRITE`)을 기본 적용한다.
- 권장 순서
  - 락 획득 -> 검증 -> 차감/반영 -> 저장
- 예외 발생 시 전체 롤백을 보장한다(부분 반영 금지).

## 9. 식별자/번호 생성 규칙
- `max+1` 방식 금지(동시성 충돌 위험).
- PK 확보 후 비즈니스 번호를 생성한다.
- 예: `SALE-{yyyyMMdd}-{id 5자리 zero-pad}`
- 포맷 생성 로직은 서비스 내부 전용 helper로 둔다.

## 10. Repository 규칙
- 도메인 쿼리는 Repository 메서드명으로 의도를 드러낸다.
- 락 쿼리는 메서드명에 의도를 반영한다.
  - 예: `findBySkuIdAndLocationIdForUpdate`
- 목록 조회 성능용 배치 조회 메서드를 준비한다.
  - 예: `findAllBySaleHeaderIdIn`

## 11. DB/마이그레이션 규칙
- 인덱스/제약/DDL은 SQL 마이그레이션 파일로 관리한다.
- 핵심 인덱스는 v1부터 반영한다(MVP라도 최소 인덱스 유지).
- 엔티티 자동 생성에 의존하지 않고, 운영 기준은 마이그레이션을 단일 진실원으로 삼는다.

## 12. 예외/에러코드 규칙
- `BaseResponseStatus`에 도메인별 코드 대역을 할당한다.
- 메시지는 한국어 기준으로 일관되게 관리한다.
- 대표 실패 케이스(없음/검증실패/재고부족 등)를 명시적으로 분리한다.

## 13. 주석/문서화 규칙
- 주석은 한국어 기준으로 작성한다.
- 깨진 주석/문자 인코딩이 보이면 즉시 복구 후 다음 작업 진행한다.
- 구현 완료 시 `docs/codex/...` 하위에 계획/구현 문서를 남긴다.

## 14. 인코딩/파일 수정 규칙
- 인코딩 안전을 위해 코드 수정은 `apply_patch` 우선 사용한다.
- 대량 치환/스크립트 수정 후에는 해당 파일 재열람으로 깨짐 여부를 확인한다.
- UTF-8(BOM 여부는 기존 프로젝트 기준 준수)로 일관 유지한다.

## 15. 네이밍/스타일 규칙
- 메서드명은 의도 중심으로 명확하게 작성한다.
- Req/Res 네이밍은 기능 접두어를 붙여 혼동을 줄인다.
- 불필요한 추상화는 줄이고, 협업자가 바로 읽히는 구조를 우선한다.

## 16. 테스트 기준(도메인 공통)
- 정상 흐름: 생성/목록/상세 정합성 검증
- 예외 흐름: 필수값 누락/수량 오류/조회 실패/재고 부족 검증
- 동시성: 동일 자원 동시 요청에서 경합 안전성(oversell 방지) 검증
- 식별자: 번호 포맷 및 유일성 검증
